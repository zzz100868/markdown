# 面经：Arya

## 项目简介（简历可用，1～2 句）

基于 Vue 3、Vite、Pinia 和 Vditor 的纯前端在线 Markdown 编辑器，支持多文档本地管理、所见即所得/分屏编辑、本地 Markdown 导入，以及 PNG、PDF、DOCX、PPT 多格式导出。项目重点解决无后端场景下的富文本状态同步、本地持久化、导出渲染链路和第三方静态资源自托管问题。

## 简历 bullet（4～6 条）

- 设计 Pinia + Storage Adapter 的多文档持久化方案，将文档元数据与正文内容拆分存储，支持旧版单文档数据自动迁移，并为 IndexedDB 或云端同步预留扩展边界。
- 集成 Vditor 编辑器并接管其缓存策略，通过初始化时序控制、1 秒防抖保存、切换前强制保存和删除兜底，保证编辑器黑盒状态与应用状态一致。
- 实现本地 Markdown 导入与多文档侧边栏管理，覆盖新建、重命名、删除、切换、移动端侧栏收起等核心交互，提升纯前端编辑器的可用性。
- 搭建 PNG、PDF、DOCX、PPT 纯前端导出链路，分别基于 html2canvas、html2pdf.js、html-to-docx 和 Reveal.js 完成渲染与下载，导出稳定性数据（待测）。
- 编写 `prestart` / `prebuild` 前置脚本自动同步 Vditor 与 DOCX vendor 静态资源，降低外部 CDN 依赖和版本漂移风险，构建体积收益（待测）。

## 面试问题（按频率排序）

### Q1（频率：高）[前端/架构] 这个项目的整体架构是什么，为什么定位成纯前端应用？

**口播版（第一人称，精简口播）**
我做这个项目，核心就是让用户打开浏览器就能编辑 Markdown，不用登录、不用后端。用 Vue 3 做组件层，Vite 构建，Vditor 当编辑器内核，Pinia 管状态，localStorage 默认存储。复杂度拆成三层：Main.vue 集成编辑器，docStore 管业务状态，LocalStorageAdapter 管持久化，导出页路由懒加载隔离。结果零后端可部署，保留多文档和导出能力，后续换 IndexedDB 或云端也有清晰替换点。

**追问 1**：纯前端架构最大的风险是什么？

**口播版（精简口播）**：最大风险是浏览器环境有硬边界。localStorage 容量小、同步写入可能卡主线程，不同浏览器对导出库兼容性也有差异。我的做法是用 Adapter 封装存储，业务层只调接口；导出拆到独立路由页；关键交互加兜底，比如 DOM 没渲染完时提示稍后再试。结果适合个人知识管理，如果变大量文档或团队协作，优先升级存储层。

**追问 2**：如果让你重构成前后端协同，你会怎么演进？

**口播版（精简口播）**：不会推翻前端结构，沿着存储适配器边界逐步替换。新增 RemoteStorageAdapter，保持 docStore 方法不变；服务端设计文档元数据表、版本表和冲突策略，比如用 updatedAt 或版本号做乐观并发控制；再加登录态和同步状态提示。结果前端大部分可复用，风险集中在同步协议和冲突处理，不用重写整个应用。

### Q2（频率：高）[前端/状态] 你是怎么做多文档本地持久化的？

**口播版（第一人称，精简口播）**
核心场景是多文档比单文档复杂，列表渲染、正文读取、删除切换、旧数据迁移都要处理。我的方案是元数据和正文分离：`arya_documents` 存列表和标题，正文按 `arya_doc_<id>` 独立存储，当前激活文档用 `arya_active_doc`。行动上 LocalStorageAdapter 封装操作，docStore 暴露业务方法。结果侧边栏只依赖轻量列表，切换时才读正文，也方便迁移 IndexedDB。

**追问 1**：为什么不把所有文档存在一个 JSON 里？

**口播版（精简口播）**：所有文档塞一个 JSON，小数据时简单，但文档多了性能和一致性就崩了。侧边栏只要标题和时间，一个 JSON 同时包含几百篇正文，每次列表刷新都要解析大量无关内容。我的做法是列表当索引，正文按 ID 单独存，保存时只写当前文档 key。结果读写成本分离，文档越多收益越大。

**追问 2**：localStorage 数据损坏或旧版本数据怎么处理？

**口播版（精简口播）**：当成存储层必须兜底的异常处理。localStorage 可能被清理、插件改坏，老版本只存了单文档 key。行动上 `getDocsList` 用 `try/catch` 包 `JSON.parse`，失败返回空列表；`migrateFromLegacy` 检查旧 key，迁移成"未命名文档"并删除旧 key。结果启动更稳，缓存坏了也能回到空文档状态，旧数据迁移集中在 Adapter 内。

### Q3（频率：高）[前端/编辑器] Vditor 这种第三方编辑器黑盒，你怎么和 Pinia 状态同步？

**口播版（第一人称，精简口播）**
Vditor 有内部状态，Pinia 也维护当前文档，两边都当状态源就容易覆盖。我的做法明确单一持久化真相：关闭 Vditor `cache.enable`，应用层接管保存；初始化时在 `after` 回调把 `store.currentContent` 用 `setValue` 填入；输入时 1 秒防抖；切换文档前先强制保存，再读取新文档 `setValue`。结果 Vditor 管编辑体验，Pinia 和 Adapter 管业务状态与持久化，边界清楚。

**追问 1**：为什么初始化时要关注 `after` 回调和 `manualSetCount`？

**口播版（精简口播）**：第三方编辑器初始化不是同步的，DOM 存在但实例不一定 ready，直接 `setValue` 可能失败或被后续初始化覆盖。我的做法是先尝试立即 `setValue`，失败交给 `after` 回调兜底；用 `manualSetCount` 记录是否已注入，避免重复覆盖。结果初始化路径兼容性更好，避免异步 ready 的竞态。

**追问 2**：如果用户快速切换文档，会不会丢内容？

**口播版（精简口播）**：快速切换有专门保护，防抖会延迟写入，刚输入完就切文档单靠定时器可能丢内容。我的做法是让切换成为一致性边界：离开前必须保存最新值。行动上 `onSelectDoc` 第一步调用 `saveCurrentDoc`，清掉未执行的防抖定时器，直接读取 `vditor.getValue()` 保存；保存完成后再 `selectDoc` 并回填。结果快速切换时上一文档最新内容不依赖 1 秒定时器，比只做输入防抖更可靠。

### Q4（频率：高）[前端/性能] 为什么要做防抖保存？这个策略有什么取舍？

**口播版（第一人称，精简口播）**
编辑器输入很频繁，localStorage 是同步 API，每次写入占主线程。持续打字时每键都写缓存，输入体验和页面响应会变差。我的做法定义 `SAVE_DEBOUNCE_MS = 1000`，输入时先清上一次定时器，用户停止输入后保存；切换文档和组件卸载前立即保存。结果常规写入次数显著减少，关键边界又能及时落盘。适合本地轻量编辑，大文档需进一步监控。

**追问 1**：1 秒这个时间怎么验证是否合适？

**口播版（精简口播）**：1 秒是工程初值，不是绝对正确。时间太短写入仍频繁，太长则增加崩溃丢内容窗口。我的做法是用可观测方式验证：准备不同大小文档，模拟连续输入 30-60 秒，统计写入次数、耗时、输入帧率和内容一致性；测试切文档、刷新、关闭标签页时的立即保存。预期 1 秒能降低写入压力，用户几乎感知不到延迟；大文档卡顿则考虑增量保存或 IndexedDB。

**追问 2**：防抖保存失败时如何兜底？

**口播版（精简口播）**：失败可能来自容量不足、隐私模式限制、localStorage 抛异常或编辑器不可用。当前切换文档和卸载前强制保存一次，减少定时器未执行的丢失；读取侧有异常兜底。若加强，`saveDocContent` 增加 `try/catch` 和错误返回，容量不足通过 message 提示用户；维护内存脏标记，保存失败时禁止静默切换，提示导出或复制。结果失败从隐性丢失变成显性可处理。

### Q5（频率：高）[前端/交互] 删除当前文档、空列表和首次启动这些边界怎么处理？

**口播版（第一人称，精简口播）**
多文档编辑器边界场景多，删除正在编辑的文档、本地没文档、迁移后没激活文档。我的任务是不出现白屏或空指针，让用户始终有可编辑入口。行动上启动时 `docStore.init` 先迁移旧数据并读列表，没激活文档但有列表就选第一篇；列表为空 `Main.vue` 创建"未命名文档"。删除时 Adapter 删除正文 key，当前文档被删时 active 指到下一篇或清空；组件层 `onDocDeleted` 没下一篇时创建新文档。结果删除、首次打开、迁移后都能闭环。

**追问 1**：为什么 store 和组件层都处理删除后的下一篇？

**口播版（精简口播）**：存储层知道 localStorage 的 active key，删当前文档后应修正底层 active 状态，保证缓存自身一致。组件层知道 Vditor 实例和 UI，需要把新正文回填或没文档时新建空文档。Adapter 更新 key 和列表，store 刷新响应式状态，`Main.vue` 调用 `applyVditorContent` 同步编辑器。结果两处处理职责不同：一处保数据不坏，一处保界面可用。

**追问 2**：这些兜底会不会掩盖真正的错误？

**口播版（精简口播）**：兜底确实有风险，要区分"用户可恢复场景"和"程序异常"。首次启动没文档、用户删最后一篇是正常业务分支，自动创建空文档合理；保存失败、JSON 解析失败、导出库加载失败不能只静默兜底。改进是在 `catch` 分支增加日志和用户提示，比如提示"本地文档索引损坏，已进入空文档模式"，提供导出或清理缓存入口。结果体验连续，真正风险不沉默。

### Q6（频率：中高）[前端/导出] PNG、PDF、DOCX、PPT 导出链路分别怎么实现？

**口播版（第一人称，精简口播）**
用户希望 Markdown 转成不同格式，项目没有后端。我的做法完全依赖浏览器能力和前端库：复用 `PreviewVditor` 渲染预览 DOM；PNG 用 `html2canvas` 截取预览区；PDF 用 `html2pdf.js` 转 DOM；DOCX 动态加载 `html-to-docx.browser.js`，拿预览 HTML 拼完整文档；PPT 用 Reveal.js 的 Markdown 插件，`---` 和 `--` 表示横向和纵向幻灯片。结果不同格式独立，无需上传服务端，隐私和部署成本友好。

**追问 1**：为什么 DOCX 要动态加载脚本？

**口播版（精简口播）**：DOCX 库比较重，不是每个用户都用。直接打进首页主包，会让所有用户为少数导出动作付出加载成本。我的做法是在 `ExportDocx.vue` 里通过 `loadHTMLtoDOCX()` 动态插入 `/vendor/html-to-docx.browser.js`，加载前补齐 `Buffer` 和 `global` 兼容对象；构建前置脚本复制 npm 包浏览器文件到 `public/vendor`。结果首页更轻，DOCX 依赖只在导出页加载；代价是验证 vendor 文件存在和加载失败提示。

**追问 2**：导出最容易出问题的地方是什么？

**口播版（精简口播）**：最容易出问题的是 DOM 渲染结果和目标文件格式之间的差异。Markdown 里可能有图片、SVG、表格、代码高亮、长文档分页、跨域资源；浏览器里看正常，转 Canvas、PDF、DOCX 后未必一致。我的做法是把问题拆到各格式链路分别兜底：PNG 导出前 SVG 转 Canvas，PDF 忽略预览操作栏，DOCX 拼完整 HTML 和基础样式，PPT 遵循 Reveal.js 分隔符规则。结果常见内容可覆盖，复杂文档导出质量需回归测试，量化数据待测。

### Q7（频率：中高）[前端/工程化] 为什么要把 Vditor 和 vendor 静态资源复制到 public？

**口播版（第一人称，精简口播）**
项目依赖 Vditor 样式、字体和运行时静态资源，也依赖 DOCX 浏览器脚本。运行时从外部 CDN 拉资源，离线部署、内网环境或 CDN 波动都会影响可用性。我的做法是在 `package.json` 的 `prestart` 和 `prebuild` 里自动复制 `node_modules/vditor/dist` 到 `public/vditor/dist`，也复制 `@turbodocx/html-to-docx` 浏览器脚本到 `public/vendor`；运行时 Vditor 配置 `cdn: "/vditor"`。结果开发和构建前自动刷新资源，部署产物更自包含，排查 404 更直接。

**追问 1**：这种复制脚本有什么风险？

**口播版（精简口播）**：风险在构建可重复性和依赖路径稳定性。第三方包内部目录改名，`cpSync` 路径会失效；CI 环境没正确安装依赖，`prebuild` 直接失败；public 残留旧资源可能造成版本错配。我的做法是让失败尽早暴露：脚本先创建目录，复制前清理旧的 `public/vditor`，确保每次来自当前依赖版本；后续给构建日志增加明确错误提示，CI 里跑一次 `npm run build`。结果风险集中在构建阶段，比运行时从 CDN 随机失败更可控。

**追问 2**：如果上生产，你会怎么验证资源自托管有效？

**口播版（精简口播）**：上生产前做两类验证：构建产物验证和运行时验证。先运行 `npm run build`，检查 `dist/vditor` 和 `dist/vendor` 是否存在；再用浏览器 Network 面板过滤外部域名，确认 Vditor 字体、主题、脚本都从本项目路径加载；最后断网或用本地静态服务器打开页面，测试编辑器初始化、预览、DOCX 导出。结果如果通过就说明自托管链路有效；有 404 就回 `prebuild` 路径和 Vite public 拷贝规则排查。

### Q8（频率：中）[前端/路由] 导出页为什么单独做路由和懒加载？

**口播版（第一人称，精简口播）**
导出功能依赖库比较杂，比如 `html2canvas`、`html2pdf.js`、Reveal.js 和 `html-to-docx`，用户进编辑器首页不一定会立刻导出。我的做法是让核心编辑路径尽量轻：把导出功能拆成 `/export/png`、`/export/pdf`、`/export/docx`、`/export/ppt`、`/export/html` 等子路由，在 `src/router/routes/main.js` 里用动态 `import()` 懒加载；导出页共享 `Frame.vue` 外壳和 `HeaderNav`，正文通过当前文档内容渲染预览。结果首页加载更聚焦编辑器和文档管理，导出代码按需加载，各格式错误隔离更清楚。

**追问 1**：导出页如何拿到当前文档内容？

**口播版（精简口播）**：导出页拿内容做了两层读取。页面刚创建时 Pinia 可能还没完成初始化，如果完全等异步 store，可能出现短暂空白。我的做法是在导出页 `data()` 里先同步读取 `arya_active_doc` 和 `arya_doc_<id>` 作为兜底；`mounted` 后再调用 `store.init()`，用 `store.currentContent` 覆盖 `pdata`。结果首屏预览更快，正式数据由统一 store 兜底。代价是 key 规则在导出页有重复，后续可抽成 helper 降低维护成本。

**追问 2**：如果导出页和主编辑器内容不一致怎么办？

**口播版（精简口播）**：先看用户进入导出页前当前内容是否已保存。主编辑器输入是防抖保存，跳转导出时如果没触发立即保存，导出页从 localStorage 读到的可能是旧内容。我的做法是导出命令触发前通知主编辑器执行一次 `saveCurrentDoc()`，或把当前内容放 Pinia 由导出页优先读 store；同时用测试覆盖"输入后立刻导出"场景。结果导出内容与编辑器内容一致，避免生成旧版本文件。

### Q9（频率：中）[前端/兼容] 移动端和响应式方面你处理了什么？

**口播版（第一人称，精简口播）**
项目不是只给桌面端，平板或手机也可能用。桌面端可同时展示侧边栏和分屏预览，移动端空间有限，照搬桌面布局会遮挡编辑区。我的做法是在 `Main.vue` 里用 `window.innerWidth <= 960` 判断移动端，移动端默认收起侧边栏，展开时加遮罩；编辑器区域根据侧边栏状态动态调整 margin；Vditor 预览在移动端默认不分屏，给编辑区留更多空间。`HeaderNav.vue` 也根据 768px 阈值隐藏标题和全屏按钮。结果移动端优先保证写作和文档切换，复杂导出和全屏更偏桌面。

**追问 1**：为什么移动端默认关闭分屏预览？

**口播版（精简口播）**：移动端默认关闭分屏预览，因为空间是主要约束。Vditor 分屏同时占编辑区和预览区，375px 或 414px 宽度下输入区域过窄，键盘弹出后体验更差。我的做法是 Vditor 的 `preview.show` 根据 `!this.isMobile` 控制移动端不默认展示预览，侧边栏也默认收起减少遮挡。结果移动端主路径更直接：打开、输入、保存、切换文档；用户仍可通过编辑模式或导出页查看渲染结果。牺牲即时预览换可编辑性。

**追问 2**：响应式判断用 resize 有什么注意点？

**口播版（精简口播）**：使用 resize 关注两个问题：监听清理和状态突变。组件切换后不移除监听会内存泄漏或重复回调；每次宽度变化都强行展开侧边栏会打断用户选择。我的做法是 `Main.vue` 和 `HeaderNav.vue` 在 `mounted` 添加 resize 监听，`beforeUnmount` 移除；`Main.vue` 的 `onResize` 在桌面端如果已收起侧边栏就保持收起，移动端默认收起避免遮挡。结果监听生命周期干净，布局变化温和。后续可考虑 CSS media query 或 `matchMedia` 简化部分 JS 判断。

### Q10（频率：中）[前端/事件] 你为什么用了 mitt 事件总线，而不是所有东西都放 Pinia？

**口播版（第一人称，精简口播）**
Pinia 负责长期业务状态，比如文档列表、当前文档 ID、当前正文；但有些动作是一次性通知，比如 HeaderNav 导入本地 Markdown 后需要告诉主编辑器刷新 Vditor 内容。我的做法是区分长期状态和瞬时事件，使用 mitt 作为轻量事件总线，通过全局 `$eventBus` 发送 `reload-content`，`Main.vue` 在 `mounted` 时订阅，收到后调用 `applyVditorContent` 回填编辑器；组件销毁时取消订阅。结果导入完成这种"一次性命令"没污染 Pinia 状态，store 保持文档数据中心职责，组件间协作也比较直接。

**追问 1**：事件总线会不会让数据流变乱？

**口播版（精简口播）**：事件总线滥用会让数据流难追踪，因为发送方和接收方没有显式父子关系。我的原则只用它处理瞬时通知，不承载业务真相。导入文件后真正的文档内容已通过 docStore 写进 store 和 localStorage，事件总线只是通知 Vditor 刷新显示。做法上 `reload-content` 不携带正文，接收方自己读 `store.currentContent`。结果事件只触发动作，不保存状态；即使事件丢失，底层数据仍然正确，最多 UI 需手动刷新。

**追问 2**：不用事件总线还有什么方案？

**口播版（精简口播）**：不用事件总线有几个方案。第一种把导入逻辑上提到 `Main.vue`，父组件直接调用编辑器刷新，显式但让 HeaderNav 失去独立性。第二种用 Pinia 增加 `contentVersion`，导入后递增，`Main.vue` watch 刷新，数据流统一但 store 会多一个偏 UI 的状态。第三种通过 `provide/inject` 暴露刷新方法。我选择 mitt 因为当前只有少量一次性通知，成本最低。结果代码简单，但会控制使用范围；如果事件数量增加，会重新评估是否改成 store token 或更明确的组合式函数。

### Q11（频率：中）[前端/安全与隐私] 纯前端编辑器在安全和隐私上有什么考虑？

**口播版（第一人称，精简口播）**
项目处理用户输入的 Markdown 文档，可能包含私人笔记、代码片段或工作内容。上传到后端或第三方接口隐私风险高；浏览器本地处理也要注意导入、图片和渲染内容的安全边界。我的做法是在功能实现上尽量减少不必要的数据外发：文档内容保存在 localStorage，本地 Markdown 导入用 FileReader 读取不上传；图片插入当前实现为 data URL 直接写入 Markdown，避免依赖外部图床；导出也在本地 DOM 上完成。结果用户内容默认留在浏览器环境内，部署不需要后端存储。但如果 Markdown 渲染开放 HTML，需关注 XSS 和 Vditor 安全配置。

**追问 1**：data URL 图片有什么缺点？

**口播版（精简口播）**：data URL 图片隐私好、无需上传、离线可用，但缺点也明显。图片转 Base64 后体积膨胀，写进 Markdown 正文会让 localStorage 很快接近容量上限，也会让编辑器渲染、保存和导出变慢。我的做法是在隐私和容量间做取舍，当前选择 data URL 是为避免第三方图床依赖和上传风险，适合少量图片的个人编辑场景。结果实现简单且本地闭环；要支持大量图片，会改成 IndexedDB Blob 存储或可选上传服务，正文里只保存引用 URL，同时提供容量提示和图片压缩策略。

**追问 2**：Markdown 渲染可能带来什么安全问题？

**口播版（精简口播）**：安全问题主要来自 HTML 注入和外部资源加载。用户可能粘贴含有 `<script>`、事件属性或恶意 iframe 的内容，渲染器没过滤就形成 XSS；外部图片或链接也可能暴露访问行为。我的做法是检查 Vditor 对 HTML、sanitize、链接 target 和资源加载的配置，必要时关闭危险 HTML 或增加白名单；导出时避免把操作栏、脚本节点写进目标文件。结果编辑器保留 Markdown 表达力，不会把所有 HTML 当可信内容。当前项目这块还可补充专门的安全回归用例。

### Q12（频率：中）[前端/可扩展] 如果要把存储从 localStorage 换成 IndexedDB，你怎么做？

**口播版（第一人称，精简口播）**
当初做 Adapter 的目的就是让存储可替换。localStorage 简单但容量小、同步阻塞，不适合大量长文档和图片；IndexedDB 容量更大、异步 API 更适合富文本数据。我的做法是在不大改组件和业务 store 的情况下替换底层存储：新增 `IndexedDBStorageAdapter`，实现和 `IStorage.js` 一致的方法；然后把导出的单例从 `localStorageAdapter` 切到新适配器，或通过工厂按配置选择。结果 `docStore` 和 `Main.vue` 调用方式基本不变，迁移风险集中在 Adapter 内，回归重点在保存、切换、删除和导出读取这些数据入口。

**追问 1**：IndexedDB 迁移时怎么保证旧数据不丢？

**口播版（精简口播）**：迁移时要保证旧数据不丢，当成可恢复流程，不是启动时随便搬一次。用户本地可能已有 localStorage 文档，直接切 IndexedDB 旧内容可能看不到。我的做法是在新 Adapter 的 `migrateFromLegacy` 里读取 `arya_documents` 和每个 `arya_doc_<id>`，逐篇写入 IndexedDB；写完后再读取数量和内容摘要做校验；只有校验成功才标记迁移完成。结果迁移可重复、可中断恢复，失败时能回退读取 localStorage。清理旧数据可延迟到用户确认或多个版本后。

**追问 2**：换成 IndexedDB 后组件层完全不用改吗？

**口播版（精简口播）**：理论上组件层大部分不用改，但不会承诺完全零改动。IndexedDB 是异步 API，当前 store 方法已经是 async，替换很友好；但导出页目前在 `data()` 阶段同步读取 localStorage 做兜底，这部分需要调整。我的做法是先用搜索找所有 `localStorage.getItem` 直接调用，尤其是各导出页；把它们改成通过 store 或统一 helper 异步读取。结果主编辑器和 store 改动很小，导出页兜底逻辑需重做。这也说明 Adapter 边界越严格，后续扩展成本越低。

### Q13（频率：中）[前端/测试] 这个项目你会怎么做测试和质量保障？

**口播版（第一人称，精简口播）**
质量风险集中在状态一致性、浏览器兼容和导出结果。普通单元测试很难覆盖 Vditor 真实 DOM、文件导入、Canvas 导出和路由跳转。我的做法是建立分层测试策略：先给 `LocalStorageAdapter` 和 `docStore` 写单元测试，覆盖新建、保存、重命名、删除、迁移和损坏 JSON；再用 Playwright 做端到端测试，模拟输入、切换文档、删除当前文档、导入 Markdown、进入导出页；最后对导出功能做人工加自动结合的回归，至少验证按钮可用、文件生成触发、控制台无错误。结果核心数据链路可自动验证，复杂导出链路也有明确回归清单。当前仓库测试还可补强。

**追问 1**：最应该优先测哪个链路？

**口播版（精简口播）**：最优先测"输入保存再切换文档"这条链路，直接关系用户数据安全。编辑器产品最严重的问题不是按钮样式错，而是内容丢了。我的做法是测试创建两篇文档，第一篇输入后立刻切第二篇，再切回断言内容仍存在；再刷新页面检查 active 文档和正文是否正确；测删除当前文档后是否切到下一篇或新建空文档。结果只要这条链路稳定，项目基本可信度就有保障，之后再扩展导出和响应式测试。

**追问 2**：导出功能怎么自动化验证？

**口播版（精简口播）**：导出功能自动化要分层，不一定一开始就解析所有文件内容。PNG、PDF、DOCX、PPT 依赖浏览器渲染和下载行为，完全精确比对成本较高。我的做法是先验证可用性，再逐步验证质量：Playwright 监听下载事件，点击导出按钮后确认生成对应扩展名文件，检查控制台无异常；PNG 可进一步读取图片尺寸，PDF/DOCX 可用解析库检查基本文本是否存在；PPT 预览可检查 Reveal DOM 是否生成 slide。结果先覆盖"不报错且能下载"，再覆盖"内容基本正确"，最后用固定样例做视觉或文本快照。复杂样式仍建议保留人工验收清单。

### Q14（频率：中）[前端/问题排查] 如果用户反馈内容偶尔被覆盖，你会怎么定位？

**口播版（第一人称，精简口播）**
内容被覆盖当成状态竞态问题处理。可能发生在初始化时 `setValue` 重复执行、切换文档时旧保存晚于新内容回填、导入文件后事件刷新顺序错乱，或防抖定时器在不合适时机执行。我的做法是找到"谁最后写入了错误内容"：给 `saveCurrentDoc`、`debouncedSave`、`applyVditorContent`、`selectDoc`、`saveContent` 增加临时日志，记录 `activeDocId`、内容摘要、时间戳和调用来源；然后复现快速输入、快速切换、导入后立即编辑等路径。结果如果发现旧定时器晚写，在切换和回填前清理定时器；如果是初始化重复覆盖，就收紧 `manualSetCount` 和 ready 状态判断。

**追问 1**：为什么记录内容摘要，而不是完整内容？

**口播版（精简口播）**：记录内容摘要而不是完整内容，主要考虑隐私和日志可读性。Markdown 可能包含敏感信息，直接打印完整正文到控制台或监控会制造安全问题；定位覆盖问题通常只需知道内容版本是否变化、长度是否异常、是否为空。我的做法是记录 `activeDocId`、内容长度、前几个字符的安全截断、hash 或时间戳，再结合调用栈和操作步骤判断哪次写入覆盖了哪次。结果既能定位竞态，又不会把用户文档泄露到日志里。正式环境里这类日志还应默认关闭或做脱敏。

**追问 2**：如果是防抖导致旧内容晚保存，怎么修？

**口播版（精简口播）**：如果确认是防抖定时器晚保存，修复重点放在生命周期边界。用户在文档 A 输入后触发定时器，马上切到文档 B，如果定时器回调里没绑定正确的文档 ID，就可能把旧内容写到当前 active 文档。我的做法是让每次延迟保存都明确保存对象：一种做法是在设置定时器时捕获当时的 `activeDocId` 和内容，回调里调用 `saveDocContent(docId, content)`；另一种是切换文档前彻底清理定时器并立即保存当前内容，当前项目已经采用这个方向。结果异步回调不再依赖变化中的全局 active 状态，覆盖风险明显下降。

### Q15（频率：中）[通用/项目表达] 这个项目你最想向面试官强调的工程亮点是什么？
